# 前端安全实战：XSS与CSRF防护完全指南

> *2026-03-19 | 前端安全 | 预计阅读 12 分钟*

## 概述

在2026年的前端开发生态中，安全问题比以往任何时候都更加重要。随着单页应用(SPA)的普及和前后端分离架构的成熟，前端开发者需要承担更多的安全责任。本文将深入探讨两种最常见且危险的白帽攻击：XSS（跨站脚本攻击）和CSRF（跨站请求伪造），并提供2026年最佳防护实践。

## 一、XSS（跨站脚本攻击）

### 1.1 攻击原理

XSS 攻击的核心是：**攻击者将恶意脚本注入到网页中，当用户访问该页面时，脚本在用户浏览器中执行**。这可能导致：
- 窃取用户 Cookie、Session
- 盗用用户身份进行操作
- 篡改页面内容
- 键盘记录、屏幕截图
- 传播恶意软件

### 1.2 XSS 类型

#### 反射型 XSS
恶意脚本通过 URL 参数传递，服务器未经处理直接反射给客户端。

```
https://example.com/search?q=<script>alert('xss')</script>
```

#### 存储型 XSS
恶意脚本被存储到数据库，每次访问都会触发。

#### DOM 型 XSS
不经过服务器，直接在客户端 JavaScript 中操作 DOM 导致。

### 1.3 防护策略（2026）

#### 1.3.1 输入验证与过滤

```javascript
// 使用 DOMPurify 净化 HTML（2026年推荐方案）
import DOMPurify from 'dompurify';

function sanitizeUserInput(dirty) {
  return DOMPurify.sanitize(dirty, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'br'],
    ALLOWED_ATTR: ['href', 'title', 'target']
  });
}

// 在 React 中的使用
function CommentBox({ content }) {
  const cleanContent = sanitizeUserInput(content);
  return <div dangerouslySetInnerHTML={{ __html: cleanContent }} />;
}
```

#### 1.3.2 Content Security Policy（CSP）

```nginx
# Nginx 配置 CSP（推荐 v3 版本）
add_header Content-Security-Policy "
  default-src 'self';
  script-src 'self' https://trusted.cdn.com 'nonce-{random}';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' https://api.example.com;
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';
" always;
```

**2026年 CSP 最佳实践**：
- 使用 `nonce` 而非 `unsafe-inline`
- 避免使用 `'unsafe-eval'`
- 启用 `frame-ancestors 'none'` 防御点击劫持
- 对于 API 请求，使用 `connect-src` 严格限制

#### 1.3.3 React/Vue 内置防护

React 默认对插值进行转义：

```jsx
// 安全：自动转义
const userInput = '<script>alert("xss")</script>';
<div>{userInput}</div> // 输出为纯文本，不会执行

// 危险：必须手动净化
<div dangerouslySetInnerHTML={{ __html: userInput }} />
```

Vue 3 同样自动转义：

```vue
<template>
  <!-- 安全 -->
  <div>{{ userContent }}</div>
  
  <!-- 危险，需要 v-html 配合净化 -->
  <div v-html="sanitizedContent"></div>
</template>
```

#### 1.3.4 HTTP Only Cookie

防止通过 JavaScript 窃取 Cookie：

```javascript
// 后端设置
res.cookie('sessionId', token, {
  httpOnly: true,   // 不可被JS访问
  secure: true,     // 仅HTTPS传输
  sameSite: 'strict' // 防止CSRF
});
```

## 二、CSRF（跨站请求伪造）

### 2.1 攻击原理

CSRF 攻击的核心是：**诱导用户在已登录的网站执行非预期的操作**。

攻击流程：
1. 用户登录 A 网站，获得 Cookie
2. 用户访问恶意网站 B
3. B 网站发出请求到 A 网站（携带 Cookie）
4. A 网站误认为是用户真实操作

### 2.2 防护方案

#### 2.2.1 SameSite Cookie 属性（首选）

```javascript
// Set-Cookie: sessionId=xxx; SameSite=Strict; Secure; HttpOnly
```

- `Strict`: 完全禁止第三方请求携带 Cookie（最安全）
- `Lax`: 允许安全的 HTTP 方法（GET）携带（2026年推荐）
- `None`: 不限制（需配合其他方案）

**注意**：某些浏览器（Safari）默认使用 `Strict`，需要注意兼容性。

#### 2.2.2 CSRF Token

```javascript
// 1. 服务端生成 Token 并注入到页面
app.use((req, res, next) => {
  const token = crypto.randomBytes(32).toString('hex');
  req.csrfToken = token;
  res.locals.csrfToken = token;
  next();
});

// 2. 前端在表单中提交
<form action="/transfer" method="POST">
  <input type="hidden" name="_csrf" value="{{ csrfToken }}">
  <button type="submit">转账</button>
</form>

// 3. Axios 自动携带
axios.defaults.xsrfCookieName = 'XSRF-TOKEN';
axios.defaults.xsrfHeaderName = 'X-CSRF-Token';

// 4. Vue/React 中使用
axios.post('/api/transfer', data, {
  headers: {
    'X-CSRF-Token': getCookie('XSRF-TOKEN')
  }
});
```

#### 2.2.3 Double Submit Cookie

```javascript
// 前端生成随机值，同时放在 Cookie 和请求头中
const csrfToken = crypto.randomUUID();
document.cookie = `csrf=${csrfToken}`;

// 后续请求携带
fetch('/api/action', {
  method: 'POST',
  headers: {
    'X-CSRF-Token': csrfToken
  }
});

// 后端验证两者是否一致
if (req.headers['x-csrf-token'] !== req.cookies.csrf) {
  return res.status(403).json({ error: 'CSRF验证失败' });
}
```

### 2.3 2026年 CSRF 防护最佳实践

**单页应用（SPA）推荐方案**：
```javascript
// 统一请求拦截器
axios.interceptors.request.use(config => {
  if (['POST', 'PUT', 'DELETE', 'PATCH'].includes(config.method.toUpperCase())) {
    const token = getCookie('XSRF-TOKEN');
    if (token) {
      config.headers['X-CSRF-Token'] = token;
    }
  }
  return config;
});
```

**配合 SameSite=Lax**：
```javascript
// 服务端设置
app.use(session({
  cookie: {
    httpOnly: true,
    secure: true,
    sameSite: 'lax', // 使用 Lax 而非 None
    maxAge: 24 * 60 * 60 * 1000
  }
}));
```

## 三、其他常见安全威胁

### 3.1 点击劫持（Clickjacking）

**防御**：
```html
<!-- 设置 X-Frame-Options 或 CSP frame-ancestors -->
<meta http-equiv="X-Frame-Options" content="DENY">
<!-- 或 -->
Content-Security-Policy: frame-ancestors 'self' https://trusted.com
```

### 3.2 中间人攻击（MITM）

**防御**：
- 强制 HTTPS（HSTS）
- 使用 Subresource Integrity (SRI)

```html
<script src="https://cdn.example.com/lib.js"
        integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8wC"
        crossorigin="anonymous"></script>
```

### 3.3 依赖供应链攻击

**防护措施**：
- 使用 npm audit 或 yarn audit 定期扫描
- 锁定版本（package-lock.json / yarn.lock）
- 使用安全扫描 CI（如 Snyk, GitHub Dependabot）

```bash
# 定期检查
npm audit --audit-level=moderate
yarn audit

# 自动修复
npm audit fix
```

### 3.4 敏感信息泄露

**不要在前端存储**：
- API 密钥
- 数据库连接字符串
- 加密盐值

**使用环境变量**（通过构建时注入）：
```javascript
// .env 文件（不提交到仓库）
VITE_API_URL=https://api.example.com
VITE_APP_KEY=xxx

// 使用
console.log(import.meta.env.VITE_API_URL);
```

## 五、实战案例：安全的 REST API 设计

```javascript
// server.js (Node.js + Express)
const express = require('express');
const helmet = require('helmet');
const rateLimit = require('express-rate-limit');
const csrf = require('csurf');
const cookieParser = require('cookie-parser');

const app = express();

// 1. 安全头
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'", "'nonce-${nonce}'"],
      styleSrc: ["'self'", "'unsafe-inline'"],
      imgSrc: ["'self'", "data:", "https:"],
      connectSrc: ["'self'", "https://api.example.com"]
    }
  },
  hsts: {
    maxAge: 31536000,
    includeSubDomains: true,
    preload: true
  }
}));

// 2. 速率限制
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15分钟
  max: 100, // 限制每个IP 100次请求
  message: '请求过于频繁，请稍后再试'
});
app.use('/api/', limiter);

// 3. 解析与验证
app.use(express.json({ limit: '10kb' }));
app.use(cookieParser());

// 4. CSRF 保护（仅对状态变更请求）
app.use(csrf({ cookie: { httpOnly: true, sameSite: 'lax' } }));

// 5. 认证中间件
const authenticate = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: '未授权' });
  
  try {
    const payload = jwt.verify(token, process.env.JWT_SECRET);
    req.user = payload;
    next();
  } catch (err) {
    res.status(401).json({ error: '令牌无效' });
  }
};

// 6. 受保护的路由
app.post('/api/transfer', authenticate, (req, res) => {
  // 业务逻辑
  const { amount, to } = req.body;
  
  // 输入验证
  if (typeof amount !== 'number' || amount <= 0) {
    return res.status(400).json({ error: '金额无效' });
  }
  
  // 执行转账...
  res.json({ success: true });
});

// 7. 错误处理（不泄露细节）
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ error: '服务器内部错误' });
});
```

## 六、前端安全清单（开发自检）

### 开发阶段
- [ ] 使用 TypeScript 或 PropTypes 进行类型检查
- [ ] 所有 API 请求使用 HTTPS
- [ ] 敏感信息不存储在 localStorage/sessionStorage
- [ ] 第三方库从官方源安装
- [ ] 定期更新依赖（npm outdated）

### 构建阶段
- [ ] 配置 CSP 头
- [ ] 设置 HSTS
- [ ] 启用压缩（避免信息泄露）
- [ ] 移除 source map（或仅限内网）

### 部署阶段
- [ ] 禁用不必要的 HTTP 方法（如 TRACE, PUT）
- [ ] 配置 WAF（Web应用防火墙）
- [ ] 日志记录安全事件
- [ ] 定期进行安全扫描

### 运维阶段
- [ ] 监控异常访问（大量 404、暴力破解）
- [ ] 定期轮换密钥和令牌
- [ ] 备份与恢复演练
- [ ] 遵循最小权限原则

## 七、总结

前端安全是一个持续的过程，而非一次性的任务。在2026年，随着 Web 技术的演进，攻击手段也在不断更新。开发者需要：

1. **保持警惕**：关注 OWASP Top 10 最新变化
2. **深度防御**：多层防护，不依赖单一手段
3. **自动化**：将安全检查集成到 CI/CD
4. **持续学习**：跟进最新的安全漏洞与修复方案

**记住**：安全不是功能，而是基础。每一个看似微小的疏忽，都可能成为系统崩溃的起点。
